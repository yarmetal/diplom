# План тестирования мобильного приложения «Мобильный хоспис»

**Версия:** 1.0 

**Дата составления:** 08.06.2026  

---

## 1. Введение

Цель документа — описание стратегии, объёма, методов и этапов тестирования мобильного приложения «Мобильный хоспис». Документ предназначен для команды тестирования, разработчиков и руководителя дипломной работы.

---

## 2. Объект тестирования

**Приложение:** «Мобильный хоспис»  
**Платформа:** Android (версия API 24+ / Android 7.0+)  
**Основной функционал:**
- Авторизация пользователей.
- Просмотр и управление новостями (создание, редактирование, удаление, фильтрация, сортировка).
- Просмотр тематических цитат на вкладке «Love is all».
- Информационный раздел «О приложении» (политика конфиденциальности, пользовательское соглашение).

**Требования к тестированию:**  
Приложение тестируется в соответствии с описанными в дипломной работе сценариями и нефункциональными требованиями (удобство интерфейса, корректность работы на типовых устройствах).

---

## 3. Критерии начала и окончания тестирования

### 3.1. Критерии начала
- Предоставлена сборка приложения (APK) с версией, фиксированной в дипломной работе.
- Сформирована тестовая среда (эмулятор Android API 24+ или реальное устройство).
- Утверждён чек-лист и тест-кейсы.
- Настроены инструменты автоматизации (Android Studio, Espresso, Gradle).

### 3.2. Критерии окончания
- Выполнены все тест-кейсы из раздела 6 (ручная проверка) и автотесты (где применимо).
- Зафиксированы все обнаруженные дефекты, критические и блокирующие — исправлены.
- Составлены отчёты о ручном и автоматизированном тестировании.
- Отчётные документы переданы руководителю дипломной работы.

---

## 4. Виды тестирования

| Вид тестирования | Описание |
|------------------|-----------|
| Позитивное | Валидные данные и корректные сценарии (успешная авторизация, создание новости со всеми полями и т.д.). |
| Негативное | Невалидные данные (пустые поля, спецсимволы, неверные логин/пароль, будущие даты). |
| Функциональное | Проверка соответствия работы всех модулей заявленным требованиям. |
| UX-тестирование | Удобство навигации, понятность интерфейса, доступность элементов управления. |
| GUI-тестирование | Соответствие макетам: расположение, цвета, шрифты, адаптация под разные размеры экранов (в рамках эмулятора). |

## 5. Подход к тестированию и автоматизация

1. **Ручное тестирование** — выполнение чек-листа и части тест-кейсов (помечены как **ручные**).  
2. **Автоматизация** — с помощью Espresso + JUnit для сценариев, где это целесообразно (указано «автоматизируется»).  
3. **Оценка покрытия** — на основе созданного чек-листа.  
4. **Отчётность** — Allure для автотестов, сводная таблица для ручных проверок.

## 8. Риски и их минимизация

| Риск | Вероятность | Влияние | Способ минимизации |
|------|-------------|---------|--------------------|
| Изменение идентификаторов (id) элементов интерфейса | Средняя | Высокое | Использовать в автотестах теги `testTag`, ресурсы, а не абсолютные id. |
| Переименование, добавление или удаление разделов приложения | Низкая | Высокое | Регулярно синхронизировать тесты с актуальной версией приложения, поддерживать чек-лист. |
| Сложность поиска уникальных id для автоматизации | Средняя | Среднее | Запрашивать у разработчика добавление атрибутов `testId` или использовать комбинации `withParent()`. |
| Обнаружение критических дефектов, блокирующих прохождение части сценариев | Средняя | Высокое | Приоритизировать тесты; дефекты документировать и информировать руководителя. |
| Недостаток времени (90 часов) из-за неожиданных проблем | Средняя | Среднее | Вести учёт времени, сократить ручное тестирование за счёт автоматизации (объёмы указаны как ручные). |

## 10. Состав отчётных документов

1. **План тестирования** (настоящий документ).  
2. **Чек-лист** (XLSX / Markdown) с отметками о выполнении.  
3. **Тест-кейсы** (в виде таблицы с шагами и ожидаемым результатом — отдельный файл или приложение к плану).  
4. **Отчёт о ручном тестировании** (количество пройденных/не пройденных тестов, найденные дефекты).  
5. **Отчёт о выполненных автотестах** (`allure-results.zip` с отчётом о результатах прогона тестов (Allure)).  
6. **Результат** (файл `Result.md` с результатом сравнения времени проверки приложения по чек-листу руками и UI-тестами)
